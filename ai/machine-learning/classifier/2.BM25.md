BM25 should be the next one after TF-IDF because they look similar, but their **best business roles are different**.

Simple conclusion first:

```
TF-IDF is mainly used as:
text → vector → classifier

BM25 is mainly used as:
query → retrieve/rank relevant documents or sentences
```

So BM25 is usually not your final classifier.
 It is more useful as a **search / retrieval / evidence engine**.

------

# 1. BM25 is not a classifier either

BM25 is not a classification model.

It does not learn:

```
customer text → high intent
customer text → complaint risk
customer text → price sensitive
```

BM25 answers a different question:

```
Given a query, which documents/sentences are most relevant?
```

Example:

```
Query:
"refund complaint money back"

Customer message database:
1. "I want my money back."
2. "Can I get a discount?"
3. "The delivery was delayed."
4. "I want to complain about the service."

BM25 ranks:
1. "I want my money back."
2. "I want to complain about the service."
```

So BM25 is closer to:

```
search engine ranking
evidence retrieval
RAG retrieval
historical case search
```

not pure classification.

------

# 2. Core principle of BM25

BM25 is based on three ideas:

```
1. Important words should increase relevance.
2. Repeating a word too many times should not increase relevance linearly.
3. Very long documents should not automatically win just because they contain more words.
```

This is the key difference from simple keyword counting.

------

## 2.1 Word matching

If the query contains:

```
refund
complaint
money back
```

and a customer message also contains these terms, BM25 gives it a higher score.

So BM25 is still lexical.

It mainly depends on word overlap.

------

## 2.2 Term frequency saturation

Suppose two messages:

```
A: "refund refund refund refund refund"
B: "refund"
```

A should score higher than B, but not 5x higher.

Why?

Because after a word appears several times, additional repetitions provide less new information.

This is called **frequency saturation**.

In business terms:

```
A customer mentioning "refund" 3 times is stronger than 1 time.
But mentioning it 50 times should not make the score 50x larger.
```

This is very reasonable for customer-service search.

------

## 2.3 Document length normalization

Long documents naturally contain more words.

If you search:

```
refund
```

a 200k-word customer history is more likely to contain “refund” somewhere than a short message.

But that does not mean the whole customer is more relevant.

BM25 corrects for document length.

This matters a lot for your data because customer histories can be very long.

Without length normalization, long customers may always rank high simply because they contain more words.

BM25 tries to avoid that.

------

# 3. Input and output

## Input

BM25 needs:

```
1. A query
2. A set of documents / messages / chunks
```

Example query:

```
refund complaint money back chargeback
```

Documents can be:

```
customer messages
conversation chunks
support tickets
CRM notes
call summaries
product reviews
```

## Output

BM25 outputs:

```
ranked documents / chunks / sentences
```

Example:

```
{
  "query": "refund complaint money back",
  "top_results": [
    {
      "text": "I want my money back.",
      "score": 12.4
    },
    {
      "text": "I will complain if nobody solves this.",
      "score": 10.1
    }
  ]
}
```

So BM25 output is not a class label.

It is:

```
relevance ranking
```

------

# 4. BM25’s advantages compared with other methods

## 4.1 Compared with TF-IDF: BM25 is usually better for search

TF-IDF can also be used for retrieval, but BM25 is usually stronger for search because it handles:

```
term frequency saturation
document length normalization
```

Example:

```
Query:
refund
```

Document A:

```
"refund refund refund refund refund"
```

Document B:

```
"I want a refund because the service failed."
```

Simple TF-IDF may over-reward repeated words.
 BM25 is more balanced.

So:

```
TF-IDF = better as classifier features
BM25 = better as search/ranking method
```

In your system:

```
TF-IDF:
train a refund-risk classifier

BM25:
find the exact refund-related messages
```

They solve different parts.

------

## 4.2 Compared with embeddings: BM25 is better for exact words

Embeddings are good for semantic similarity.

But for business risk, exact words matter.

Example words:

```
refund
chargeback
lawsuit
complaint
allergy
medical accident
contract termination
invoice
```

If these words appear, you often want to catch them exactly.

BM25 is very good at that.

Embedding models may sometimes smooth the meaning and retrieve semantically similar but less exact results.

For example, if the query is:

```
chargeback
```

BM25 will strongly prefer documents with the exact word `chargeback`.

Embeddings may retrieve broader payment-dispute content, which can be useful, but may miss the exact precision you want.

So BM25 is suitable when:

```
exact term matching matters
evidence needs to be literal
risk keywords must not be missed
```

------

## 4.3 Compared with LLMs: BM25 is cheap and controllable

An LLM can read a customer history and find evidence.

But for 200k words per customer, this is expensive and slow.

BM25 can first retrieve the most relevant chunks:

```
customer history
→ BM25 retrieves top 20 suspicious chunks
→ LLM reads only those chunks
→ LLM judges risk / summarizes evidence
```

This is much cheaper than:

```
LLM reads entire 200k-word history
```

So BM25 is very useful before LLM.

Its role is:

```
reduce search space
find candidate evidence
lower LLM cost
increase controllability
```

------

## 4.4 Compared with pure rules: BM25 is more flexible

A rule may say:

```
if text contains "refund", mark refund risk
```

But BM25 allows richer queries:

```
refund money back cancel order chargeback complaint dissatisfied
```

It can rank results by relevance instead of only true/false matching.

So rules answer:

```
Does this keyword appear?
```

BM25 answers:

```
Which messages are most relevant to this risk theme?
```

That is more useful for evidence retrieval.

------

# 5. BM25’s disadvantages

## 5.1 BM25 does not understand meaning deeply

BM25 still depends on lexical overlap.

Query:

```
refund
```

Customer says:

```
I want my money back.
```

If your query does not include `money back`, BM25 may miss it.

Embeddings handle this better because they understand semantic similarity.

So BM25 is weak when customers use many different expressions for the same concept.

Example:

```
refund
money back
reimbursement
reverse the transaction
return the payment
cancel and repay
```

BM25 needs you to include more synonyms in the query.

Embeddings can generalize better.

------

## 5.2 BM25 is not a business classifier

BM25 can retrieve refund-related messages.

But it cannot reliably decide:

```
Is this customer truly complaint-risk?
Is this customer high purchase intent?
What should sales do next?
```

Because retrieval is not classification.

Example:

```
"I do not need a refund."
```

BM25 may retrieve it for query `refund`.

But the meaning is negative.

So BM25 gives candidate evidence, but another layer must judge the meaning.

That layer can be:

```
rules
classifier
cross-encoder
LLM
human review
```

------

## 5.3 BM25 struggles with implicit intent

Customer says:

```
Can I come this weekend?
Can two people use the same package?
Is installment payment available?
```

A human may infer:

```
high purchase intent
```

But BM25 will not know this unless your query contains similar words:

```
appointment
weekend
installment
package
```

Even then, it does not really infer intent.

So BM25 is weak for:

```
implicit intent
business reasoning
customer psychology
conversion probability
next action recommendation
```

------

## 5.4 BM25 depends heavily on tokenization

For Chinese text, tokenization matters a lot.

Example:

```
我想退款
```

Good tokenization:

```
我 / 想 / 退款
```

Bad tokenization:

```
我 / 想 / 退 / 款
```

If tokenization is wrong, BM25 search quality drops.

For your business, you need a custom dictionary:

```
退款
退货
客诉
转化
到店
复购
私域
种植牙
正畸
脱毛
高意向
低意向
价格敏感
```

Without domain tokenization, BM25 may perform badly.

------

## 5.5 BM25 can retrieve noisy evidence

If your query is broad:

```
price discount expensive cheap package
```

BM25 may retrieve many messages that mention price but are not truly price-sensitive.

Example:

```
"The price is okay."
```

This contains `price`, but does not mean the customer is price-sensitive.

So BM25 needs:

```
careful query design
negative pattern handling
reranking
classifier or LLM validation
```

------

# 6. When BM25 is suitable

## Scenario 1: Evidence retrieval

This is BM25’s strongest business scenario.

If the model says:

```
complaint_risk = high
```

business users will ask:

```
Why?
Which sentence proves it?
```

BM25 can retrieve relevant messages.

Example query:

```
refund complaint dissatisfied money back delay service attitude
```

Output:

```
{
  "evidence": [
    "I want my money back.",
    "The service has been delayed many times.",
    "I will complain if this is not solved."
  ]
}
```

Why suitable?

Because evidence often contains explicit words. BM25 is excellent at finding literal matches.

------

## Scenario 2: RAG before LLM judgment

For long customer histories, do not send everything to an LLM.

Better:

```
customer history chunks
→ BM25 retrieves top 20 relevant chunks
→ embedding retrieval retrieves semantic chunks
→ reranker merges results
→ LLM reads only selected evidence
→ LLM produces final judgment
```

Why suitable?

Because BM25 cheaply reduces the amount of text.

This is especially important for:

```
200k-word customer histories
long support records
multi-month customer conversations
large CRM notes
```

------

## Scenario 3: Search historical cases

Sales or customer service may ask:

```
Find customers who mentioned chargeback.
Find conversations about refund failure.
Find cases similar to delivery delay complaints.
Find customers who compared competitor brands.
```

BM25 is suitable because these are search tasks.

Output is not one label. Output is a ranked list of relevant records.

------

## Scenario 4: Risk and compliance investigation

For risk control, exact terms matter.

Example:

```
lawsuit
complaint
refund
false advertising
allergy
medical accident
contract dispute
chargeback
```

BM25 can retrieve documents containing these risk signals.

Why suitable?

Because for risk review, missing exact words can be costly.

But BM25 should not be the only layer.

Better:

```
keyword rules
+ BM25 retrieval
+ classifier
+ LLM/manual review
```

------

## Scenario 5: Building training data

BM25 can help collect candidate examples for labeling.

Example:

You want to build a refund-risk classifier.

Use BM25 queries:

```
refund money back chargeback cancel order complaint
```

Retrieve top conversations, then manually label them.

This gives you a high-density candidate set.

Why suitable?

Because random sampling may produce very few refund cases if refund risk is rare.

BM25 helps you find likely positive examples faster.

------

# 7. When BM25 is not suitable

## Not suitable 1: Final customer classification

BM25 can retrieve relevant text, but it does not fully classify.

Example:

```
"I do not want a refund."
```

BM25 may retrieve it for `refund`, but the meaning is opposite.

So BM25 alone is not enough for:

```
refund risk classification
high intent classification
complaint risk classification
price sensitivity classification
```

Use BM25 as evidence, then classifier/LLM as judge.

------

## Not suitable 2: Deep semantic similarity

If users express the same idea in many ways, BM25 may miss them.

Example:

```
refund
money back
reverse transaction
reimbursement
cancel payment
```

Unless all terms are in the query, BM25 may not connect them.

Use embeddings for semantic recall.

------

## Not suitable 3: Next-action recommendation

BM25 cannot decide:

```
Should I send discount?
Should I call?
Should I wait?
Should I send case proof?
```

It can retrieve text, but cannot reason over the whole customer state.

For next action, use:

```
classification scores
conversion model
customer behavior features
rules
LLM strategy generation
uplift model
```

------

## Not suitable 4: Customer-level long-term profiling by itself

BM25 can retrieve relevant chunks, but it does not summarize a customer’s whole behavior.

For example, it cannot directly output:

```
{
  "customer_type": "high-intent price-sensitive",
  "conversion_probability": 0.72,
  "best_action": "send limited discount and follow up today"
}
```

It needs to be part of a larger system.

------

# 8. BM25 compared with other methods

## BM25 vs TF-IDF

BM25 is better for:

```
search
retrieval
ranking
evidence finding
```

TF-IDF is better for:

```
feature extraction for classifiers
cheap supervised classification
interpretable word features
```

Practical combination:

```
TF-IDF + LinearSVC:
predict label

BM25:
retrieve evidence supporting that label
```

------

## BM25 vs embeddings

BM25 advantages:

```
exact keyword matching
cheap
fast
interpretable
good for rare terms
good for risk words
```

Embedding advantages:

```
semantic similarity
synonym handling
implicit meaning
similar customer retrieval
cross-wording generalization
```

Best combination:

```
BM25 retrieves exact keyword matches.
Embeddings retrieve semantic matches.
Reranker/LLM judges final relevance.
```

------

## BM25 vs LLM

BM25 advantages:

```
cheap
fast
deterministic
good at finding literal evidence
scales to huge corpora
```

LLM advantages:

```
understands meaning
handles negation
summarizes
reasons
generates next action
```

Best combination:

```
BM25 finds candidate evidence.
LLM reads candidate evidence and reasons.
```

------

## BM25 vs rules

Rules are binary:

```
contains refund → yes/no
```

BM25 is ranked:

```
which messages are most relevant to refund risk?
```

Rules are good for hard triggers.
 BM25 is better for ranked retrieval.

Best combination:

```
rules for must-catch risk terms
BM25 for broader evidence retrieval
```

------

# 9. How BM25 should be used in your customer-dialogue system

For your data, BM25 should not index one whole customer as one document.

Do not do:

```
customer 200k words → one BM25 document
```

Better:

```
customer history
→ split into messages / sessions / time windows
→ BM25 index over chunks
```

Then for each label, prepare query templates.

Example:

```
Refund risk query:
refund money back chargeback cancel order complain dissatisfied

Price-sensitive query:
discount expensive cheaper budget coupon price package

High-intent query:
appointment book schedule available today tomorrow deposit

Competitor comparison query:
compare competitor other store better cheaper difference
```

Then:

```
label query
→ retrieve top-k chunks
→ classifier/LLM verifies
→ aggregate to customer-level label
```

This gives you:

```
classification + evidence
```

not only a black-box score.

BM25 can produce outputs like:

```
{
  "query": "refund complaint money back",
  "top_evidence": [
    {
      "customer_id": "C001",
      "message": "I want my money back.",
      "score": 14.2
    },
    {
      "customer_id": "C001",
      "message": "I will complain if this is not solved.",
      "score": 11.8
    }
  ]
}
```

In a full business system, BM25 helps produce:

```
{
  "customer_label": "complaint_risk",
  "risk_score": 0.81,
  "evidence": [
    "I want my money back.",
    "I will complain if this is not solved."
  ],
  "review_required": true
}
```

Notice:

```
BM25 provides evidence.
Another model provides the final label.
```

------

# 11. Final conclusion for BM25

BM25 is not outdated.
 It is outdated only if you expect it to understand meaning like an LLM or embedding model.

Its real value is:

```
fast, cheap, accurate lexical retrieval
```

For your customer-dialogue system, BM25 is best used as:

```
evidence retrieval
RAG pre-filter
historical case search
risk keyword investigation
training data mining
```

It should not be used alone as:

```
final classifier
customer profiler
reasoning engine
next-action recommender
```

The correct position is:

```
TF-IDF = cheap classification feature
BM25 = evidence/search layer
BGE-M3 = semantic retrieval/classification layer
LightGBM/CatBoost = business feature fusion layer
LLM = hard-case reasoning and explanation layer
```

Next one should be **Logistic Regression**, because after TF-IDF/BM25, you need to understand the first real classifier: why a simple linear model can still be very strong for business text classification.
